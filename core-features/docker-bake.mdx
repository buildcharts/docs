BuildCharts generates a `docker-bake.hcl` file and uses [Docker Buildx Bake](https://docs.docker.com/reference/cli/docker/buildx/bake/) as the execution layer for the build plan.

## What Bake is

Docker describes Bake as a declarative way to define build configuration, instead of rebuilding complex `docker build` commands from CLI flags on every run.

In BuildCharts, that means:
- `build.yml` defines build intent
- `Chart.yaml` maps target types to OCI-hosted implementations
- `buildcharts generate` renders the final `docker-bake.hcl`
- `docker buildx bake` executes the generated plan

## How BuildCharts uses Bake

BuildCharts uses Bake in the following ways:

- Declarative build configuration in a file instead of long CLI commands
- A native way to run multiple targets in parallel
- Groups and targets that map cleanly to BuildCharts stage types
- A structured place for args, tags, outputs, contexts, entitlements, provenance, and SBOM settings
- A standard Docker-native interface that works locally and in CI

### Typical flow

```bash
buildcharts generate
docker buildx bake --file .buildcharts/docker-bake.hcl
```

### Advanced Docker build features

Bake exposes first-class support for features that matter in BuildCharts:
- Outputs for build artifacts
- Multi-platform builds
- Advanced cache configurations
  - Docker layer caching with `cache-from` and `cache-to`
  - Docker registry caching, where published images and build cache are stored in a registry
  - Remote cache backends such as `s3` and `azblob`
- Deduplication of shared context transfer
- Remote builders through the `kubernetes` driver
- Dependencies between targets through target contexts
- Entitlements
- Provenance and SBOM attestation
- Target overrides with `--set`

### Parallel targets

Bake runs specified targets in parallel. In BuildCharts, a repository can generate separate `build`, `test`, `nuget`, and `docker` targets from one metadata model.

### Specific targets

Run a single target to execute one stage. For example, to run only tests:

```bash
docker buildx bake --file .buildcharts/docker-bake.hcl test
```

To run test and nuget:

```bash
docker buildx bake --file .buildcharts/docker-bake.hcl test nuget
```


### Better reuse of shared configuration

Bake groups related targets and centralizes shared settings. BuildCharts uses that to emit common variables and target-level configuration without duplicating command-line flags per step.

### Easier inspection and debugging

Bake supports commands such as:
- `docker buildx bake --print`
- `docker buildx bake --list`
- `docker buildx history ls`
- `docker buildx history inspect`

That makes it easier to inspect the generated build plan and troubleshoot failures after execution.


## SBOM and provenance

BuildCharts executes the generated `docker-bake.hcl` through Docker Buildx, so provenance and SBOM attestations use the standard Buildx attestation flow.

To inspect the generated SBOM locally before you push an image, override the generated `docker` target and use a local output:

```bash
docker buildx bake \
  --file .buildcharts/docker-bake.hcl docker \
  --set docker.attest=type=sbom \
  --set docker.output=type=local,dest=.buildcharts/output/sbom
```

This writes the SBOM to `.buildcharts/output/sbom/sbom.spdx.json`.
