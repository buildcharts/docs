---
title: "Using outputs"
description: "Understand where BuildCharts writes exported artifacts and how to consume them."
---

BuildCharts writes outputs based on the target type it generates into the Bake plan.

## Output behavior by stage type

| Stage type | Output behavior |
| --- | --- |
| `build` | `type=cacheonly` |
| `docker` | `type=docker` |
| Any other type | Export to `.buildcharts/output/<type>` |

That means `test`, `nuget`, and any custom non-docker type export local files under `.buildcharts/output`.

## Consume test outputs

The shipped test sample reads TRX files from `.buildcharts/output/test`.

```bash
buildcharts generate
docker buildx bake --file .buildcharts/docker-bake.hcl
```

After the build finishes, inspect:

```text
.buildcharts/output/test
```

## Consume NuGet outputs

NuGet package stages export artifacts to:

```text
.buildcharts/output/nuget
```

This is the same pattern used by the `samples/dotnet-nuget` example.

## Work with Docker outputs

`docker` stages use `type=docker`, so the image is loaded into your Docker environment instead of being exported to `.buildcharts/output/docker`.

Use tags in `build.yml` so the result is easy to reference:

```yaml
targets:
  src/App/App.csproj:
    - type: docker
      with:
        base: mcr.microsoft.com/dotnet/runtime:10.0
        tags: ["docker.io/example/app:${VERSION}-${COMMIT}"]
```

## Generate a build summary

`buildcharts summary` exports:
- `.buildcharts/output/SUMMARY.md`
- `.buildcharts/output/buildcharts.dockerbuild`

It also appends the summary to `GITHUB_STEP_SUMMARY` on GitHub Actions.

## Keep outputs stable in scripts

If you script around exported files, reference the type folder directly:

```powershell
Get-ChildItem .buildcharts/output/nuget -Filter *.nupkg
```

```powershell
& "$PSScriptRoot\..\Parse-Trx.ps1" ".buildcharts/output/test"
```
